# S3-compatible backend (AWS S3, R2, B2, MinIO, Wasabi, Spaces, Storj, …)

One claude-mirror backend works against every S3-compatible service. The differentiator is the `s3_endpoint_url` field in the YAML — leave it empty for AWS S3 proper, set it to the provider's S3 endpoint otherwise.

Supported out of the box (anything that speaks the S3 API works; this list is non-exhaustive):

- **AWS S3** — the original
- **Cloudflare R2** — S3 API with zero egress fees
- **Backblaze B2 (S3 API)** — cheap durable storage
- **Wasabi Hot Cloud Storage**
- **MinIO** — self-hosted, S3-compatible (commonly run on a NAS / homelab box)
- **Tigris** — globally distributed S3-compatible object store
- **IDrive E2** — predictable-pricing S3 API
- **Linode Object Storage** — Akamai's S3 service
- **DigitalOcean Spaces** — S3 API with built-in CDN
- **Storj DCS** — distributed-cloud S3 gateway
- **Hetzner Storage Box** — when used through the S3 gateway
- Any other service that exposes a conformant S3 API

The same `boto3>=1.34` SDK ships in the base install — no extras flag, no separate provider-specific package. The SDK is lazy-imported inside `claude_mirror/backends/s3.py` so users who never touch S3 pay zero startup cost.

## Quick start — AWS S3

### Step 1: Create a bucket

In the [AWS S3 console](https://s3.console.aws.amazon.com/s3/), create a bucket. Pick a region close to you. Block Public Access can stay enabled — claude-mirror only ever uses authenticated requests.

### Step 2: Create an IAM user with the minimum permissions

Create a dedicated IAM user (or role) for claude-mirror and attach the [minimum policy](#minimum-iam-policy) below. Generate an access key + secret access key for the user — you will paste them into the YAML.

### Step 3: Run the wizard

```bash
claude-mirror init --wizard --backend s3
```

When prompted, leave `S3 endpoint URL` blank, set the bucket / region / access key ID / secret access key, accept the default prefix (the project name), and answer `n` to path-style addressing.

### Step 4: Run the doctor

```bash
claude-mirror doctor
```

The S3 deep checks confirm the bucket is reachable, list permissions work, write+delete permissions work, and the configured region matches the bucket's actual region.

## Quick start — Cloudflare R2

R2 has zero egress fees, which makes it especially attractive for personal sync.

1. In the Cloudflare dashboard, create an R2 bucket.
2. Generate an R2 API token (Access Key ID + Secret Access Key).
3. Note your account ID — it appears in the R2 endpoint URL: `https://<account_id>.r2.cloudflarestorage.com`.
4. Run the wizard:

```bash
claude-mirror init --wizard --backend s3
```

When prompted:

- `S3 endpoint URL` → `https://<your-account-id>.r2.cloudflarestorage.com`
- `S3 region` → `auto`
- `S3 bucket` → your bucket name
- access key + secret → your R2 token credentials
- path-style → `n` (R2 supports virtual-hosted style)

## Quick start — Backblaze B2 (S3 API)

B2 is one of the cheapest durable storage options at scale. Use the S3 API surface, not the native B2 API.

1. In the B2 web UI, create a bucket and an Application Key with read+write access scoped to that bucket.
2. Note the bucket's region — visible as the second segment of the S3 endpoint, e.g. `eu-central-003` for `s3.eu-central-003.backblazeb2.com`.
3. Run the wizard:

- `S3 endpoint URL` → `https://s3.<region>.backblazeb2.com` (e.g. `https://s3.eu-central-003.backblazeb2.com`)
- `S3 region` → the same region (e.g. `eu-central-003`)
- `S3 bucket` → your bucket name
- access key + secret → the B2 Application Key's `keyID` + `applicationKey`
- path-style → `n`

## Quick start — MinIO (self-hosted)

MinIO runs anywhere — laptop, NAS, homelab. The defining quirk is that MinIO requires path-style addressing.

1. On the MinIO server, create a bucket and an access key + secret.
2. Run the wizard:

- `S3 endpoint URL` → e.g. `http://nas.local:9000` (or `https://...` if you've set up TLS)
- `S3 region` → any value (`us-east-1` is conventional)
- `S3 bucket` → your bucket
- access key + secret → MinIO credentials
- path-style → `y` (this is the key MinIO-vs-everything-else difference)

## Config file reference

```yaml
backend: s3
project_path: /home/user/work/myproject

# Endpoint URL: empty for AWS S3 proper; provider URL otherwise.
s3_endpoint_url: ""

# Bucket name (must already exist).
s3_bucket: mybucket

# Region. Required for AWS proper; many S3-compat services accept any value.
s3_region: us-east-1

# Access key + secret. Leave both blank to use boto3's default credential
# chain (env vars / ~/.aws/credentials / IAM role). When blank you can
# still point claude-mirror at a specific profile via the AWS_PROFILE env var.
s3_access_key_id: AKIA...
s3_secret_access_key: ...

# Path prefix inside the bucket. Empty means "use the project name".
s3_prefix: myproject

# Path-style addressing (https://endpoint/bucket/key). Required for MinIO
# and a few S3-compat services. AWS proper accepts both styles; default
# false (virtual-hosted-style).
s3_use_path_style: false

# Polling interval in seconds for the watcher. S3 has no native push, so
# we poll list_objects_v2 to detect remote changes.
poll_interval: 30
```

## Minimum IAM policy

The smallest IAM policy claude-mirror needs is bucket-level list + object-level read/write/delete on the configured prefix:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListBucket",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::mybucket"
    },
    {
      "Sid": "ObjectRW",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:CopyObject"
      ],
      "Resource": "arn:aws:s3:::mybucket/myproject/*"
    }
  ]
}
```

Adjust `Resource` to match your `s3_bucket` and `s3_prefix`. For S3-compat providers, translate to the equivalent policy in their dashboard (R2 has bucket-scoped tokens; B2 has Application Keys scoped to a single bucket; MinIO uses standard IAM-shaped policies).

The deep doctor checks (`claude-mirror doctor --backend s3`) will tell you exactly which IAM action is missing if the policy is too tight.

## Running the doctor deep checks

```bash
claude-mirror doctor --backend s3
```

The deep-check sequence runs after the generic doctor passes:

1. **Credentials shape** — both `s3_access_key_id` and `s3_secret_access_key` are set, or both blank (default credential chain).
2. **Endpoint URL well-formed** — when set, it's a parseable URL with `http`/`https` scheme.
3. **Bucket reachable** — `head_bucket` succeeds.
4. **List permissions** — `list_objects_v2` with `MaxKeys=1` returns without 403.
5. **Write permissions** — a 1-byte sentinel `__claude_mirror_doctor_test` is `put` then `delete`d.
6. **Region consistency** — when `s3_region` is set, the bucket's actual region matches (warning, not failure).

Auth-class failures (InvalidAccessKeyId / SignatureDoesNotMatch / AccessDenied / 401) bucket: a single AUTH-bucketed line is emitted and the remaining checks short-circuit, so you see one root cause not five.

## Troubleshooting

### `InvalidAccessKeyId` or `SignatureDoesNotMatch`

The access key + secret pair is not what the provider expects. Verify both fields in the YAML; for AWS, also verify the key has not been deactivated in the IAM console.

### `NoSuchBucket`

The bucket name in `s3_bucket` does not exist on the configured endpoint. Check spelling, and on multi-region providers verify you are not pointing at the wrong region's endpoint.

### Region mismatch warnings

The advisory `Region mismatch: configured X but bucket is in Y` typically means the YAML's `s3_region` was copy-pasted from another project. Update it to match the bucket's home region; on AWS, mismatched regions can cause additional latency or 301 redirects on some operations.

### MinIO connection refuses with TLS errors

MinIO without TLS uses `http://` URLs. claude-mirror does not impose the WebDAV-style `--insecure-http` gate on S3 (boto3 itself follows whatever scheme the endpoint URL specifies). For internet-reachable MinIO instances, terminate TLS in front of MinIO (Caddy / nginx / Traefik all work) and use `https://`.

### Path-style vs virtual-hosted-style

Symptom: `Could not connect to the endpoint URL` or DNS lookup failures despite a working `curl` to the bucket. The fix is almost always toggling `s3_use_path_style: true` in the YAML. AWS / R2 / B2 / Wasabi / Spaces all work with virtual-hosted style (`https://bucket.endpoint/key`); MinIO and a few self-hosted alternatives require path-style (`https://endpoint/bucket/key`).

### Multipart upload ETags are not MD5

When boto3 uploads a file via multipart (any file >5 MiB by default), the resulting ETag is a synthetic hash with a `-N` suffix where N = number of parts. It is **not** comparable to a local MD5. claude-mirror's hash comparison falls back to size+mtime in this case, so cross-machine sync still works correctly — but if you are scripting your own integrity checks against the ETag, account for this.
